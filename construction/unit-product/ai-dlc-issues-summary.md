# AI-DLC 流程问题总结

> 本文档汇总了使用 AI-DLC 流程开发 AwSome Shop 电商系统过程中遇到的所有问题，按阶段分类整理，并附带根因分析和改进建议。

---

## 一、Inception 阶段

### 1.1 需求澄清不够充分
- **问题**：功能的具体页面、字段、规则、流程没有在需求阶段细化
- **影响**：开发阶段频繁返工，前后端对接时发现大量不一致
- **改进建议**：需求阶段应产出字段级别的功能规格说明，包括每个页面的输入/输出字段、校验规则、业务流程图

### 1.2 缺少 UI 原型
- **问题**：没有 UI 原型，对具体需求和流程缺乏直观感知，无法提前发现问题
- **影响**：前端实现与预期不符，交互逻辑缺失（如"新增商品"按钮无事件绑定）
- **改进建议**：Inception 阶段应产出低保真原型或页面线框图，明确每个页面的交互行为

### 1.3 应用设计阶段交互接口定义不清晰
- **问题**：服务间 API 契约在设计阶段定义不够精确，留到调试阶段才解决
- **影响**：auth-service 返回 `Result<AuthValidateResponse>` 嵌套结构，gateway 按 `AuthValidateResponse` 直接反序列化导致认证失败
- **改进建议**：应用设计阶段应产出完整的 API 契约文档（OpenAPI/Swagger），包括请求/响应的完整 JSON 结构

### 1.4 逆向工程对依赖分析不够深入
- **问题**：逆向工程阶段未识别各模块的 HTTP 客户端选型（WebClient vs RestTemplate）
- **影响**：删除 `spring-boot-starter-webflux` 依赖时未同步修改代码，导致编译失败
- **改进建议**：逆向工程应产出每个模块的技术依赖清单，标注关键技术选型及其使用位置

### 1.5 跨服务交互契约定义缺失
- **问题**：服务间响应格式没有统一约定
- **影响**：gateway 的 AuthServiceClient 反序列化失败（`success` 字段始终为 `false`）
- **改进建议**：设计阶段应定义统一的服务间通信契约，包括标准响应包装格式 `Result<T>`

### 1.6 认证/鉴权流程设计不完整
- **问题**：Token 的传递机制（前端存储 → 请求拦截器注入 → 后端校验）没有端到端设计
- **影响**：Dashboard 页面 401 循环跳转（请求不带 Token → 401 → 跳转登录 → 登录后又跳回 → 循环）
- **改进建议**：设计阶段应产出完整的认证流程时序图，覆盖 Token 获取、存储、注入、刷新、失效处理全链路

### 1.7 用户数据模型不完整
- **问题**：LoginResponse 缺少 `userId` 字段
- **影响**：前端无法获取当前用户 ID，积分查询等功能无法正常工作
- **改进建议**：设计阶段应明确每个 API 响应需要返回的完整字段列表，特别是登录等核心接口

## 二、Construction 阶段

### 2.1 数据库 Schema 与领域模型不匹配
- **问题**：数据库字段与领域模型设计存在不匹配，包括缺少字段、类型不一致
- **影响**：运行时数据映射错误，需要补充 Flyway 迁移脚本
- **改进建议**：代码生成阶段应自动校验 Entity/PO 字段与数据库 Schema 的一致性

### 2.2 API 路径前后端不一致
- **问题**：前端代码调用的 API 路径与后端实际定义不一致
- **影响**：前端 `productApi.create` 调 `/api/v1/admin/product/create`，后端实际路径是 `/api/v1/public/product/create`
- **改进建议**：应从统一的 API 契约文档生成前后端代码，或在构建阶段加入 API 路径一致性校验

### 2.3 代码实现缺失
- **问题**：AI-DLC 流程声称实现完成，但编译时发现大量代码未实现
- **具体表现**：
  - ProductDomainService 接口缺失方法
  - 多个 Repository 实现类（Mapper/PO/RepositoryImpl）未创建
  - 多个模块缺少 domain-api、domain-model 等 Maven 依赖
- **改进建议**：每个 unit 代码生成完成后，应强制执行编译验证步骤

### 2.4 包路径依赖错误
- **问题**：Maven 模块间依赖配置错误
- **影响**：编译失败，需要手动补充依赖
- **改进建议**：代码生成阶段应根据 import 语句自动检查并补全 Maven 依赖

### 2.5 前后端 Mock 数据残留
- **问题**：代码生成后未清理 mock 占位代码
- **具体表现**：
  - 注册页面 `handleRegister` 写着 `// TODO: 调用 authApi.register`，实际未调后端 API
  - 登录使用 MOCK_USERS，只认 `admin/admin123` 和 `employee/emp123`
  - 员工兑换记录显示 mock 数据
- **改进建议**：代码生成完成后应扫描 `// TODO`、`MOCK_` 等标记，确保无残留

### 2.6 网关路由配置不完整
- **问题**：新增的 API 路径未同步添加网关路由规则
- **影响**：category 的 public 路径 `/api/v1/public/category/**` 未加入网关路由，前端无法访问分类列表
- **改进建议**：每新增一个 API 端点，应自动检查网关路由是否覆盖

### 2.7 DDD 分层依赖违规
- **问题**：application-impl 直接依赖 cache-api，违反 DDD 架构规则
- **影响**：需要重构，将补偿逻辑封装到领域服务中
- **改进建议**：代码生成阶段应内置 DDD 分层依赖校验规则，禁止跨层直接依赖

### 2.8 Flyway 迁移脚本版本冲突
- **问题**：数据库迁移脚本版本管理混乱
- **具体表现**：
  - V2 脚本冲突，需重命名为 V3
  - V4 迁移失败（`success=0`），数据部分插入
- **改进建议**：迁移脚本版本号应纳入流程管控，生成前检查已有版本号

### 2.9 前端状态管理与后端数据不同步
- **问题**：前端 store 的数据初始化逻辑未与后端 API 对齐
- **具体表现**：
  - 登录后积分显示为 0（`user?.points ?? 0`，但登录时未设置该值）
  - 需要在 EmployeeLayout 中额外调用 API 获取积分余额
- **改进建议**：前端状态设计应明确每个字段的数据来源（登录响应 vs 独立 API 调用）

### 2.10 删除依赖未同步修改代码
- **问题**：删除 `spring-boot-starter-webflux` 依赖后，未修改使用 WebClient 的代码
- **影响**：编译失败，涉及 PointsServiceClientImpl、ProductServiceClientImpl、WebClientConfig 三个文件
- **修复方案**：将 WebClient 全部替换为 RestTemplate
- **改进建议**：删除依赖前应自动扫描代码中对该依赖的所有引用，生成影响范围报告

### 2.11 缺少自动化测试阶段和测试计划
- **问题**：AI-DLC 流程中没有设计自动化测试阶段
- **影响**：很多问题在编译或手动测试时才发现，修复成本高
- **改进建议**：每个 unit 完成后应生成并执行单元测试，集成测试应在所有 unit 完成后执行

## 三、Operations 阶段

### 3.1 本地开发环境配置缺失
- **问题**：Docker MySQL 端口映射、各服务的 `application-local.yml` 配置没有标准化文档
- **影响**：开发者搭建本地环境时需要反复试错
- **改进建议**：提供 docker-compose.yml 和标准化的本地开发环境配置指南

### 3.2 服务启动顺序依赖未文档化
- **问题**：多个微服务之间有启动顺序要求（如 auth-service 先于 gateway），但没有文档说明
- **影响**：开发者启动服务时遇到连接失败等问题
- **改进建议**：文档化服务依赖关系和推荐启动顺序，或使用健康检查 + 重试机制消除顺序依赖

### 3.3 Terraform 部署与应用配置衔接断裂
- **问题**：EKS 部署后，应用配置（数据库连接、服务发现、Redis 地址）如何从 Terraform output 传递到 K8s ConfigMap/Secret 没有设计
- **影响**：部署后需要手动配置，容易出错
- **改进建议**：Terraform output 应自动生成 K8s ConfigMap/Secret 模板

## 四、根因分析：流程问题与协作问题

> 23 个问题可以从两个维度归因：AI-DLC 流程自身的缺陷（单人开发也会出现），以及 5 人分工协作模式引起的集成问题。本项目由 5 名开发者完成 6 个 unit 的开发，Inception 阶段集体完成后，每人各自独立开发所负责的模块，最后进行集成。

### 4.1 流程问题（15 个，占 65%）

以下问题即使单人开发也会出现，属于 AI-DLC 流程缺陷或 AI 代码生成能力问题：

#### 4.1.1 缺少集成验证阶段
- **问题**：AI-DLC 流程在每个 unit 代码生成完成后没有强制的"编译 + 启动 + 冒烟测试"步骤
- **影响**：依赖缺失、接口不匹配、mock 残留等问题积累到最后才集中爆发，修复成本高
- **改进建议**：在每个 unit 的 Code Generation 完成后，增加强制的集成验证步骤：
  1. 编译验证（`mvn clean compile`）
  2. 依赖检查（import 与 Maven 依赖一致性）
  3. API 路径一致性检查（前端调用路径 vs 后端定义路径 vs 网关路由）
  4. Mock 残留扫描（`// TODO`、`MOCK_` 标记）
  5. DDD 分层合规检查

#### 4.1.2 设计与实现的一致性缺乏保障
- **问题**：设计阶段产出的文档（API 契约、数据模型、依赖关系）与实际生成的代码之间缺乏自动化校验
- **改进建议**：引入"设计-代码一致性检查"机制，在代码生成后自动比对设计文档与实际代码

#### 4.1.3 流程问题涉及的具体缺陷

| 类别 | 问题编号 | 问题 | 根因 |
|------|---------|------|------|
| 需求深度 | 1.1 | 需求澄清不够 | Inception 阶段分析深度不足 |
| 需求深度 | 1.2 | 缺少 UI 原型 | 流程中未包含原型设计环节 |
| 设计深度 | 1.4 | 逆向工程依赖分析不深 | 逆向工程阶段能力不足 |
| 设计深度 | 1.6 | 认证流程设计不完整 | 端到端流程设计缺失 |
| 代码生成 | 2.1 | Schema 与模型不匹配 | AI 生成一致性问题 |
| 代码生成 | 2.3 | 代码实现缺失 | AI 代码生成不完整 |
| 代码生成 | 2.4 | 包路径依赖错误 | AI 代码生成质量问题 |
| 架构合规 | 2.7 | DDD 分层违规 | 架构规则校验缺失 |
| 版本管理 | 2.8 | Flyway 版本冲突 | 迁移脚本版本管理机制缺失 |
| 测试 | 2.11 | 缺少自动化测试 | 流程设计缺失 |
| 运维 | 3.1 | 本地环境配置缺失 | 文档缺失 |
| 运维 | 3.2 | 启动顺序未文档化 | 文档缺失 |
| 运维 | 3.3 | Terraform 衔接断裂 | 部署流程设计问题 |

### 4.2 协作问题（8 个，占 35%）

以下问题是 5 人分工独立开发后集成时暴露的，由"分工开发 + 最后集成"的协作模式直接引起。

#### 4.2.1 服务间接口对接不一致（4 个）

| 问题编号 | 问题 | 协作根因 |
|---------|------|---------|
| 1.3 | 交互接口定义不清晰 | gateway 开发者和 auth-service 开发者对响应格式理解不同，各自开发时都能"通过"，集成时才爆 |
| 1.5 | 跨服务交互契约缺失 | auth-service 返回 `Result<AuthValidateResponse>`，gateway 按 `AuthValidateResponse` 反序列化。两个开发者对响应结构的约定不一致 |
| 2.2 | API 路径前后端不一致 | 前端开发者用 `/api/v1/admin/product/create`，product-service 开发者定义 `/api/v1/public/product/create`，没有统一的 API 契约作为"单一事实来源" |
| 2.6 | 网关路由配置不完整 | product-service 开发者新增了 category 接口，gateway 开发者不知道需要同步添加路由规则 |

#### 4.2.2 前后端数据交接不对齐（3 个）

| 问题编号 | 问题 | 协作根因 |
|---------|------|---------|
| 1.7 | 用户数据模型不完整 | 前端开发者需要 `userId` 查积分，auth-service 开发者不知道前端有这个需求 |
| 2.5 | Mock 数据残留 | 前端开发者先用 mock 数据开发 UI，后端 API 就绪后应替换，但这个"交接"没有发生 |
| 2.9 | 前端状态与后端数据不同步 | 前端假设登录响应里有积分数据，后端没有返回。各自开发时都"没问题"，集成后才发现 |

#### 4.2.3 技术选型变更协调不到位（1 个）

| 问题编号 | 问题 | 协作根因 |
|---------|------|---------|
| 2.10 | 删依赖未同步改代码 | 统一决定删 webflux，但各开发者独立开发时有人用了 WebClient。删依赖的指令下达后，order-service 的开发者 D 只删了 pom 没改代码，因为不清楚自己模块里哪些地方用了 |

#### 4.2.4 协作问题的根因分析

根因集中在三点：

1. **缺少"单一事实来源"的 API 契约** — Inception 阶段产出的接口定义是文字描述，不是机器可校验的规格（如 OpenAPI spec）。5 个人拿着同一份设计文档各自理解、各自开发，理解偏差在集成时才暴露
2. **变更影响范围不可见** — 改了一个模块的依赖或接口，不知道会影响哪些其他模块。没有跨模块的影响分析机制
3. **前后端交接点没有明确协议** — "谁返回什么字段"、"mock 什么时候替换为真实调用"这些交接没有流程保障

#### 4.2.5 针对协作模式的改进建议

| 改进措施 | 覆盖问题 | 实施阶段 |
|---------|---------|---------|
| Inception 阶段产出机器可读的 API 契约（OpenAPI spec），所有开发者基于契约生成代码骨架 | 1.3, 1.5, 2.2, 1.7 | Inception |
| 建立跨模块变更通知机制：修改公共依赖或接口时，自动识别受影响模块并通知相关开发者 | 2.10, 2.6 | Construction |
| 定义前后端交接检查清单：后端 API 就绪后，前端必须替换 mock 并验证联通 | 2.5, 2.9 | Construction |
| 每个 unit 完成后执行跨服务集成冒烟测试，而不是等所有 unit 完成后才集成 | 所有协作问题 | Construction |
