# 用户故事

## 故事组织方式

- **粒度**：细粒度 Story 级别，每个故事聚焦单一功能
- **拆分方式**：按业务领域 + 角色混合组织
- **优先级**：P0（必须实现）/ P1（应该实现）

---

## 领域 1：认证（Auth）

### US-AUTH-01：员工自行注册

**优先级**：P0
**角色**：员工

作为员工，我希望能通过用户名和密码注册账号，以便访问积分商城。

**验收标准**：

- 假设我在注册页面，当我提交有效的用户名/密码/显示名称时，则账号创建成功，我可以登录
- 假设用户名已存在，当我尝试注册相同用户名时，则收到错误提示"用户名已存在"
- 假设我提交了空字段或无效字段，当表单校验时，则显示具体的校验错误信息
- 假设注册成功，则我的角色默认设置为"employee"

### US-AUTH-02：管理员创建用户

**优先级**：P1
**角色**：管理员

作为管理员，我希望能在管理后台创建用户账号，以便为新员工或新管理员开通账号。

**验收标准**：

- 假设我在用户管理页面，当我填写用户信息并选择角色（employee/admin）时，则账号创建成功
- 假设用户名已存在，当我提交时，则显示错误信息
- 假设我不是管理员，当我尝试访问此接口时，则返回 403 禁止访问

### US-AUTH-03：用户登录

**优先级**：P0
**角色**：员工、管理员

作为用户，我希望能通过用户名和密码登录，以便获取 JWT Token 访问系统。

**验收标准**：

- 假设凭证有效，当我提交登录时，则收到 JWT Token 和用户信息（用户名、显示名称、角色）
- 假设凭证无效，当我提交登录时，则收到"用户名或密码错误"
- 假设账号已锁定，当我尝试登录时，则收到"账号已锁定，请30分钟后重试"

### US-AUTH-04：登录失败限流

**优先级**：P0
**角色**：系统

作为系统，我需要在连续5次登录失败后锁定账号，以防止暴力破解攻击。

**验收标准**：

- 假设用户已失败登录4次，当第5次失败时，则账号锁定30分钟
- 假设账号已锁定，当锁定时间到期时，则失败计数器重置
- 假设登录成功，则失败计数器重置为0
- 锁定状态存储在 Redis 中，设置 TTL

### US-AUTH-05：Token 验证（内部接口）

**优先级**：P0
**角色**：系统

作为网关服务，我需要通过 Auth 服务内部接口验证 JWT Token，以便对请求进行身份认证。

**验收标准**：

- 假设 Token 有效，当调用 POST /api/v1/internal/auth/validate 时，则返回 success=true 和 operatorId
- 假设 Token 已过期，当验证时，则返回 success=false，消息为"Token已过期"
- 假设 Token 被篡改，当验证时，则返回 success=false，消息为"Token无效"

### US-AUTH-06：获取当前用户信息

**优先级**：P0
**角色**：员工、管理员

作为用户，我希望能获取我的个人信息，以便前端展示我的姓名、角色和积分。

**验收标准**：

- 假设我已认证，当我调用用户信息接口时，则收到我的用户名、显示名称、角色
- 假设我未认证，当我调用该接口时，则返回 401

### US-AUTH-07：管理员查询用户列表

**优先级**：P1
**角色**：管理员

作为管理员，我希望能查看所有用户的分页列表，以便管理用户账号。

**验收标准**：

- 假设我是管理员，当我查询用户列表时，则收到分页的用户记录
- 假设我提供搜索关键词，当我查询时，则结果按用户名或显示名称过滤

## 领域 2：商品（Product）

### US-PROD-01：浏览商品列表

**优先级**：P0
**角色**：员工

作为员工，我希望能浏览分页的商品列表，以便找到想兑换的商品。

**验收标准**：

- 假设我在商城首页，当页面加载时，则显示已上架商品的分页网格
- 假设商品分布在多个分类中，当我选择分类筛选时，则只显示该分类的商品
- 假设我输入关键词，当我搜索时，则显示名称匹配的商品
- 仅 status=1（已上架）的商品对员工可见

### US-PROD-02：查看商品详情

**优先级**：P0
**角色**：员工

作为员工，我希望能查看商品详情，以便决定是否兑换。

**验收标准**：

- 假设我点击商品卡片，当详情页加载时，则显示名称、图片、积分价格、市场价、描述、规格、库存状态、配送方式、服务保障
- 假设商品缺货，则兑换按钮禁用，显示"库存不足"

### US-PROD-03：创建商品

**优先级**：P1
**角色**：管理员

作为管理员，我希望能创建新商品，以便员工可以兑换。

**验收标准**：

- 假设我填写了所有必填字段（名称、SKU、分类、积分价格、库存），当我提交时，则商品创建成功，状态为 status=0（未上架）
- 假设 SKU 已存在，当我提交时，则显示"SKU已存在"
- 所有字段经过校验（名称最长200字，SKU最长100字等）

### US-PROD-04：编辑商品

**优先级**：P1
**角色**：管理员

作为管理员，我希望能编辑已有商品，以便更新商品信息。

**验收标准**：

- 假设我在商品编辑页面，当我修改字段并提交时，则商品信息更新
- 假设商品不存在，则返回 404
- 乐观锁防止并发编辑冲突

### US-PROD-05：删除商品

**优先级**：P1
**角色**：管理员

作为管理员，我希望能删除商品，以便将其从目录中移除。

**验收标准**：

- 假设我点击删除商品，当我确认时，则商品被逻辑删除
- 假设商品有关联订单，删除仍然成功（逻辑删除，订单保留商品快照）

### US-PROD-06：商品上架/下架

**优先级**：P1
**角色**：管理员

作为管理员，我希望能切换商品的上下架状态，以便控制哪些商品可供兑换。

**验收标准**：

- 假设商品为未上架（status=0），当我点击"上架"时，则状态变为1，员工可见
- 假设商品为已上架（status=1），当我点击"下架"时，则状态变为0，员工不可见

### US-PROD-07：管理商品分类

**优先级**：P1
**角色**：管理员

作为管理员，我希望能管理多级分类树，以便商品组织有序。

**验收标准**：

- 假设我在分类管理页面，当我创建顶级分类时，则它出现在树的根节点
- 假设存在父分类，当我在其下创建子分类时，则它作为子节点出现
- 假设我编辑分类名称，当我保存时，则名称更新
- 假设我删除没有商品的分类，当我确认时，则分类被移除
- 假设我删除有商品的分类，则收到错误"该分类下有商品，无法删除"

### US-PROD-08：库存扣减（内部接口）

**优先级**：P0
**角色**：系统

作为订单服务，我需要扣减商品库存，以便正确追踪已兑换的商品。

**验收标准**：

- 假设库存 >= 请求数量，当调用扣减时，则库存减少，已兑换数量增加
- 假设库存 < 请求数量，当调用扣减时，则返回错误"库存不足"
- 乐观锁防止超卖

### US-PROD-09：库存恢复（内部接口）

**优先级**：P0
**角色**：系统

作为订单服务，我需要在订单取消时恢复商品库存，以便库存数据准确。

**验收标准**：

- 假设订单被取消，当调用库存恢复时，则库存增加，已兑换数量减少

## 领域 3：积分（Points）

### US-PTS-01：查询积分余额

**优先级**：P0
**角色**：员工

作为员工，我希望能查看当前积分余额，以便知道可以花多少积分。

**验收标准**：

- 假设我已认证，当我查询余额时，则显示当前余额、累计获得、累计消费
- 假设我还没有积分账户，当我查询时，则自动创建余额为0的账户

### US-PTS-02：查询积分交易明细

**优先级**：P0
**角色**：员工

作为员工，我希望能查看积分交易历史，以便追踪积分的获得和消费情况。

**验收标准**：

- 假设我有交易记录，当我查询列表时，则显示分页记录，包含类型、金额、交易后余额、描述、日期
- 假设我按类型筛选（获得/消费/退还），则只显示匹配的记录

### US-PTS-03：管理员发放积分（单个用户）

**优先级**：P1
**角色**：管理员

作为管理员，我希望能给指定员工发放积分，以便奖励员工。

**验收标准**：

- 假设我指定用户ID和金额，当我提交时，则该员工余额增加，生成一条"获得"类型的交易记录
- 假设用户不存在，则收到错误提示
- 描述字段记录发放原因

### US-PTS-04：管理员批量发放积分

**优先级**：P1
**角色**：管理员

作为管理员，我希望能同时给多个员工发放积分，以便高效分发奖励。

**验收标准**：

- 假设我提供用户ID列表和金额，当我提交时，则所有指定员工收到积分
- 假设部分用户ID无效，则有效的成功处理，无效的在响应中报告

### US-PTS-05：积分扣减（内部接口）

**优先级**：P0
**角色**：系统

作为订单服务，我需要从员工账户扣减积分，以便收取兑换费用。

**验收标准**：

- 假设余额 >= 金额，当调用扣减时，则余额减少，生成一条"消费"类型的交易记录
- 假设余额 < 金额，当调用扣减时，则返回错误"积分余额不足"
- 乐观锁防止并发超扣

### US-PTS-06：积分退还（内部接口）

**优先级**：P0
**角色**：系统

作为订单服务，我需要在订单取消时退还积分，以便恢复员工余额。

**验收标准**：

- 假设订单被取消且有积分金额，当调用退还时，则余额增加，生成一条"退还"类型的交易记录
- 退还交易记录关联原始订单

## 领域 4：订单（Order）

### US-ORD-01：兑换下单

**优先级**：P0
**角色**：员工

作为员工，我希望能使用积分兑换商品，以便获得商品。

**验收标准**：

- 假设商品已上架、库存充足、积分余额足够，当我提交兑换时，则创建订单（状态=待处理），积分被扣减，库存被扣减
- 假设商品已下架，则收到"商品已下架"
- 假设库存不足，则收到"库存不足"
- 假设积分不足，则收到"积分余额不足"
- 假设部分步骤执行后某步骤失败，则执行补偿（退还积分、恢复库存）
- 订单号生成格式：ORD + 时间戳 + 随机数

### US-ORD-02：员工查询订单列表

**优先级**：P0
**角色**：员工

作为员工，我希望能查看我的兑换订单历史，以便追踪过去的兑换记录。

**验收标准**：

- 假设我已认证，当我查询订单时，则只显示我自己的订单，分页展示
- 每条订单显示：订单号、商品名称、数量、积分金额、状态、创建时间

### US-ORD-03：管理员查询订单列表

**优先级**：P1
**角色**：管理员

作为管理员，我希望能查看所有订单，以便管理兑换流程。

**验收标准**：

- 假设我是管理员，当我查询订单时，则显示所有订单，分页展示
- 假设我按状态筛选，则只显示匹配的订单
- 假设我按订单号或用户搜索，则显示匹配的订单

### US-ORD-04：查看订单详情

**优先级**：P1
**角色**：员工、管理员

作为用户，我希望能查看订单详情，以便了解某个订单的完整信息。

**验收标准**：

- 假设订单ID有效，当我查询详情时，则显示所有订单字段，包括商品快照
- 假设我是员工，则只能查看自己的订单
- 假设我是管理员，则可以查看任何订单

### US-ORD-05：取消订单

**优先级**：P0
**角色**：员工

作为员工，我希望能取消待处理的订单，以便恢复积分和商品库存。

**验收标准**：

- 假设订单状态为"待处理"或"处理中"，当我取消时，则状态变为"已取消"，积分退还，库存恢复
- 假设订单状态为"已完成"或"已取消"，当我尝试取消时，则收到"该订单无法取消"
- 取消立即生效（无时间限制）

### US-ORD-06：管理员更新订单状态

**优先级**：P1
**角色**：管理员

作为管理员，我希望能更新订单状态，以便推进订单处理流程。

**验收标准**：

- 假设订单为"待处理"，当管理员设为"处理中"时，则状态更新
- 假设订单为"处理中"，当管理员设为"已完成"时，则状态更新
- 假设状态转换无效，则收到错误提示

## 领域 5：前端（Frontend）

### US-FE-01：前端登录对接

**优先级**：P0
**角色**：员工、管理员

作为用户，我希望前端登录连接真实的 Auth API，以便正确完成身份认证。

**验收标准**：

- 假设我输入有效凭证，当我点击登录时，则 JWT Token 被存储，我被重定向到对应角色的首页
- 假设我输入无效凭证，则显示后端返回的错误信息
- 假设 Token 过期（收到401响应），则自动跳转到登录页

### US-FE-02：前端注册页面

**优先级**：P0
**角色**：员工

作为员工，我希望能通过前端注册，以便创建我的账号。

**验收标准**：

- 假设我在登录页面，当我点击"注册"时，则显示注册表单
- 假设我提交有效信息，则账号创建成功，我可以登录

### US-FE-03：商城首页（真实数据）

**优先级**：P0
**角色**：员工

作为员工，我希望商城首页展示来自后端的真实商品，以便看到实际可兑换的商品。

**验收标准**：

- 假设后端有商品数据，当页面加载时，则展示真实商品数据
- 分类筛选标签从分类 API 获取
- 搜索输入触发后端搜索 API

### US-FE-04：商品详情页

**优先级**：P0
**角色**：员工

作为员工，我希望能查看商品详情页，以便在兑换前了解完整信息。

**验收标准**：

- 假设我点击商品卡片，当详情页加载时，则从后端获取并展示所有商品信息
- 如果库存 > 0 且我有足够积分，则显示"兑换"按钮

### US-FE-05：兑换确认流程

**优先级**：P0
**角色**：员工

作为员工，我希望兑换前有确认弹窗，以免误操作消费积分。

**验收标准**：

- 假设我点击"兑换"，则弹出确认对话框，显示商品名称、积分花费和我的当前余额
- 假设我确认，则调用兑换 API，显示成功/失败反馈
- 假设兑换失败，则显示具体错误信息

### US-FE-06：兑换记录页

**优先级**：P0
**角色**：员工

作为员工，我希望在 /orders 页面查看兑换历史，以便追踪过去的订单。

**验收标准**：

- 假设我导航到 /orders，则从后端 API 获取并展示我的订单列表
- 每条订单显示订单号、商品、积分、状态、日期

### US-FE-07：积分明细页

**优先级**：P0
**角色**：员工

作为员工，我希望在 /points 页面查看积分历史，以便了解所有积分变动。

**验收标准**：

- 假设我导航到 /points，则显示当前余额和交易列表
- 可按交易类型筛选（获得/消费/退还）

### US-FE-08：商品管理页

**优先级**：P1
**角色**：管理员

作为管理员，我希望在 /admin/products 有商品管理页面，以便进行商品增删改查。

**验收标准**：

- 假设我导航到 /admin/products，则显示带搜索/筛选的商品表格
- 可在此页面创建、编辑、删除商品，以及切换上下架状态

### US-FE-09：分类管理页

**优先级**：P1
**角色**：管理员

作为管理员，我希望在 /admin/categories 有分类管理页面，以便管理分类树。

**验收标准**：

- 假设我导航到 /admin/categories，则显示分类树形结构
- 可添加、编辑、删除分类

### US-FE-10：订单管理页

**优先级**：P1
**角色**：管理员

作为管理员，我希望在 /admin/orders 有订单管理页面，以便处理订单。

**验收标准**：

- 假设我导航到 /admin/orders，则显示所有订单，支持按状态筛选
- 可在此页面更新订单状态

### US-FE-11：积分管理页

**优先级**：P1
**角色**：管理员
